## Unresolved Disagreements

### 1. WORKFLOW.md: Always-loaded vs. on-demand skill

**Token CFO position:** Convert to on-demand skill. Saves ~900 tokens/session. Most sessions don't consult it.

**New Member position:** Keep always-loaded. It is the only file showing the full workflow picture. Making it on-demand hides the entry point from newcomers.

**Trade-off:** 900 tokens/session vs. discoverability. If `.claude/README.md` (#5) is created with a prominent link to WORKFLOW.md, the on-demand conversion becomes safer. Without the README, removing WORKFLOW.md from always-loaded context eliminates the last discoverable overview.

**Resolution path:** Implement #5 (README) first, then reassess whether WORKFLOW.md can move to on-demand in the next quarterly review.

### 2. Planning skill: Trim vs. disable

**Token CFO position:** Move to skills.nouse/ (362 lines, used only at feature start).

**Harness Evangelist position:** Keep active but trim to ~210 lines. Planning is used at the start of every significant task — disabling it means Claude loses the planning methodology when it's needed most.

**Trade-off:** ~210 tokens saved (after trim) vs. losing the planning methodology entirely when disabled. Skills in nouse/ require manual activation, which breaks the "always available" promise.

**Resolution path:** Trim first (P2 #15). If the trimmed version is still too costly after other P0-P1 savings are realized, move to nouse/ in the next review.

### 3. Worklog staleness: Hook vs. skill instruction

**Evangelist position:** Add a PostToolUse hook that reminds Claude to update worklog.doing.md at each commit.

**CFO + Quality Lead position:** The planning skill already instructs Claude to update worklogs automatically. If Claude ignores the skill instruction, it will ignore a hook reminder too. The root cause is ceremony overhead, not lack of enforcement.

**Trade-off:** A hook costs zero resting tokens but adds noise. If worklog staleness is a real problem in practice, the hook is cheap insurance. If not, it's unnecessary.

**Resolution path:** Monitor worklog freshness over the next month. If worklog.doing.md is consistently stale, add the hook. If the task classifier (#4) reduces ceremony for small tasks, staleness may self-resolve.

## Verdict

**The current .claude/ setup is helping this developer's workflow, but at unnecessarily high cost.** The core architecture — rules for always-on guidelines, skills for procedural workflows, agents for specialized subprocesses, hooks for lightweight enforcement — is sound and well-designed. The problems are evolutionary debt: the setup grew from a JavaScript project without a systematic adaptation pass, producing JavaScript artifacts in a Python template, contradictory commit conventions added at different times, and duplicated content across files that drifted apart. The document taxonomy (plan/backlog/worklog/learn/ADR) is thorough and well-conceived but imposes ceremony overhead on trivial tasks that don't need it. The proposed task size classifier (P1 #4) is the single highest-leverage fix — it preserves the full workflow for complex work while creating a fast path for the 60-70% of interactions that are small or trivial. Combined with the ~30% token reduction from trimming and consolidation (P0-P1), this setup would move from "comprehensive but costly" to "lean and effective." The quarterly review cadence documented in WORKFLOW.md should be maintained to prevent recurrence of the evolutionary debt found in this review.
