## 배치 분류

### Skills (신규 생성 권장)

반복적으로 같은 구조로 요청하는 작업 — 프롬프트로 고정하면 매번 설명 불필요.

| 스킬명 | 근거 |
|---|---|
| `streamlit-responsive` | 모바일 반응형 UI 패턴을 여러 세션에 걸쳐 반복 설계. 레이아웃 원칙 + 구현 패턴 고정 가능 |
| `openapi-review` | API 명세 설계 → 검증 → 버그 수정 패턴이 real-estate-mcp에서 반복 |
| `til-analyze` | TIL 볼트 상태 분석 → 태그 정리 → graph-health 체크가 여러 세션 반복 |
| `docker-deploy` | Docker Compose + Caddy + Cloudflare Tunnel 구성이 거의 동일 패턴으로 반복 |
| `fastapi-scaffold` | FastAPI + OAuth + MCP 초기 구성을 매번 처음부터 세팅 |

### Agents (신규 생성 권장)

여러 단계를 자율적으로 연속 처리해야 하는 작업.

| 에이전트명 | 근거 |
|---|---|
| `til-health-agent` | 볼트 전체 스캔 → 태그 통계 → 고아 노트 탐지 → 리포트 생성. 다단계 자동화 적합 |
| `deploy-check-agent` | 배포 후 컨테이너 헬스 → Caddy 라우팅 → API 엔드포인트 응답 순서 자동 검증 |
| `api-spec-validator` | OpenAPI 명세 읽기 → 실제 엔드포인트 대조 → 불일치 리포트. real-estate-mcp에서 반복 발생한 문제 |

### CLAUDE.md (보강 권장)

현재 python-template CLAUDE.md는 coding-guidelines, commit-convention, thinking-guidelines가 잘 갖춰져 있음. 아래는 **아직 빠진 패턴**:

| 지침 | 근거 |
|---|---|
| **UI 반복 수정 절제 원칙** | cosmetic 수정(색상, 위치 등)을 여러 세션에 걸쳐 조금씩 반복하는 패턴이 관찰됨. "UI는 세션 내에서 일괄 결정" 규칙 명시 필요 |
| **설계 문서 우선 원칙** | `localdocs/plan.*.md` 선행 작성 관행을 이미 실천 중이지만 문서화 안 됨. 명시적 규칙으로 격상 필요 |
| **API 명세 선 확정 원칙** | 초반 API 설계 미흡 → 후반 재설계 반복 경험. "API 변경 전 OpenAPI 명세 먼저 수정" 규칙 |
| **작업 단계 명시 (SPIKE/MVP/PROD)** | 현재 단계 구분 없이 시작 → 나중에 재설계. 시작 시 단계 선언으로 불필요한 polish 방지 |

## 작업 스타일 특징

**점진적 개선 선호:** 큰 기능 한 번에 구현보다 피드백 받고 반복적으로 다듬는 방식.

**설계 문서 선행:** 구현 전 localdocs에 설계 가이드 문서 먼저 작성 후 구현.

**문제 발견 후 재설계:** 초반 단순 구현 → 나중에 API 명세 불일치, 스키마 문제 발견 → 재검토 사이클.

**주요 기술 스택:** Django/FastAPI, React/Streamlit, Docker/Caddy/Cloudflare Tunnel, uv/pytest/ruff, OpenAPI/MCP/OAuth
